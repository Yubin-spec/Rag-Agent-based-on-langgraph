# 面试用：项目真实踩坑故事

以下故事均基于本项目实现与设计文档，面试时可任选 2～3 个讲，按「现象 → 原因 → 怎么改 → 收获」说，控制在 1～2 分钟/个。

---

## 1. 流式接口「假流式」：首字延迟跟非流式差不多

**现象**：上了 SSE 流式接口后，用户反馈「点了发送还是要等好几秒才出第一个字」，尤其是走知识库 RAG 的时候，体感和非流式差不多。

**原因**：RAG 生成阶段用的是 `ainvoke` 拿完整答案，再用 `_yield_text_chunked` 按 64 字一段往 SSE 推。本质是「先等整段生成完，再模拟流式」，首字延迟 = 检索时间 + 整段 LLM 生成时间，没有利用到模型的 token 流。

**怎么改**：单独做了流式专用路径 `_generate_grounded_answer_stream`，用 `_rag_chain.astream` 逐 token 收、边收边 yield。流式路径不再做多轮 grounding 重试（否则要缓冲整段再判，首包又变慢），只做单次生成，流结束后做一次 grounding 检查写 trace。这样首字延迟 ≈ 检索时间 + 模型首包时间。

**收获**：流式要真流式，必须从 LLM 层就用 astream；同时要明确「流式路径」和「非流式路径」在能力上的取舍（比如流式不做多轮重试），避免两边逻辑混在一起。

---

## 2. 加会话锁后，流式接口少了「恢复中断」和 15 轮校验

**现象**：为了同一会话并发写不冲突，给 `/chat` 和 `/chat/stream` 加了会话锁。上线后发现：流式接口在「会话被人工中断后恢复」、以及「超过 15 轮后的限制提示」有时不生效。

**原因**：加锁时把流式接口改成了「锁包住一个 async 生成器」，但原来在进入生成器之前做的几件事——从 DB 恢复图状态、resume 人工中断、15 轮校验——被挪到了外层或删掉了，流式路径里没再调用这些逻辑，导致锁内的流式生成器「裸跑」，没有先恢复状态、也没先判 15 轮。

**怎么改**：在 `_chat_stream_generator_impl` 内部、在拿锁之后、真正开始流式生成之前，把「从 DB 恢复状态」「resume 人工中断」「取 snapshot 数用户轮数、超 15 轮直接返回提示并 yield done」这三步重新加回去，保证流式和非流式在「状态恢复 + 轮数限制」上一致。

**收获**：重构「谁在锁内、谁在锁外」时，要对着调用链逐项核对：原来在入口做的事，拆到生成器里之后是否还在锁内、且仍在正确时机执行；最好用 checklist 或单测覆盖「恢复中断」「15 轮」等关键分支。

## 3. 同一问题高并发时，缓存没命中反而把下游打挂

**现象**：某个热点问题（比如「AEO 认证条件」）被多人同时问，监控上看到 Redis 缓存命中率不高，但 LLM 和检索的 QPS 突然飙高，延迟也上去，怀疑是「缓存没挡住」。

**原因**：缓存 key 是按问题归一化后的 hash，同一问题 key 相同。当这个 key 刚好过期或第一次被问时，大量请求同时未命中，一起回源——都去走图、检索、LLM，形成**缓存击穿**；同一时刻对同一个 key 的重复计算和下游压力叠加，把 LLM/检索打高。

**怎么改**：做了**单飞锁（single-flight）**：同一问题（同一缓存 key）未命中时，只放行一个协程回源，其它协程在按 key 分桶的锁上等；回源协程算完并写入缓存后释放锁，等待的协程再查一次缓存就能命中。同时可选**进程内 LRU** 做热 key 缓存，先查本地再查 Redis，减轻对同一 key 的集中请求。

**收获**：有缓存就要考虑击穿；高并发下「同一 key 只回源一次」是常见手段，用分桶锁既能保证语义又避免全局锁竞争。热 key 用本地一层可以进一步减压。

## 4. 数据库/向量库一抖，整站都 500

**现象**：某次 PostgreSQL 或 Milvus 抖动、连接变慢，不仅写库的接口报错，连纯对话、纯 LLM 的请求也开始大量 500，像是被「拖下水」。

**原因**：早期对 DB、Milvus 的调用没有区分**关键路径**和**非关键路径**。像对话历史持久化、问答监控上报、RAG 检索结果，一旦连接失败或超时就直接抛异常，请求整体失败。而获取连接时没有熔断，持续重试又会占满连接池/线程，把其它请求拖慢或拖挂。

**怎么改**：在 db_resilience 层做了**熔断**（连续失败 N 次后短时不再请求，冷却后半开探测），并且对每次调用区分 `critical=True/False`。关键路径（如 Text2SQL 执行必须连库）失败就抛错；非关键路径（对话历史、监控、RAG 检索、文档上传写 Milvus）失败就**降级**：返回空列表/0/跳过写、打日志，不阻塞主流程。这样 DB 或 Milvus 短暂不可用时，用户仍能对话，只是历史没持久化或检索为空。

**收获**：依赖要分「缺了就不能完成主流程」和「可以弱化或跳过」的；非关键路径统一走降级，并配合熔断，避免一个依赖把整站拉垮。

## 5. RAG 答案「看起来对」但没依据、或依据对不上

**现象**：用户反馈「回答里说的数字/条款和给的依据对不上」，或者根本没有标注依据，怀疑模型在「发挥」。

**原因**：只靠 prompt 要求「必须带 [证据N]」「只能依据参考内容」不够，模型有时仍会漏标或掺入参考外内容。没有对生成结果做**事后校验**，差的结果也会直接返回。

**怎么改**：做了两件事。（1）**答案质量校验**：用「答案与检索文档的关联度」（grounding_score）和「是否包含 [证据N]」做检查，低于阈值或缺少引用就视为可能幻觉；（2）**非流式路径多轮重试**：不达标时把「上一版关联度过低、请更严格贴合参考」和「请补充证据编号」作为 retry_note 再调一次 LLM，最多若干次，仍不达标再返回保守兜底文案。流式路径为保首字延迟只做单次生成，但在流结束后同样算 grounding 和 citation 写 trace，便于排查和后续优化。

**收获**：RAG 不能只靠 prompt 约束，要有「生成后校验 + 不达标重试或降级」的闭环；流式和非流式可以有不同的重试策略，但要都能记录质量便于分析。

## 6. Text2SQL 多表查询出来一堆重复行或错关联

**现象**：用户问「某口岸进口量和某企业名单」这类涉及多表的问题，生成的 SQL 能跑，但结果行数爆炸或关联明显不对。

**原因**：多表时模型自己「猜」关联条件，容易漏写 JOIN 条件或写错字段，变成笛卡尔积或错误关联；schema 里虽然有表结构，但表与表之间该用哪些字段关联没有显式约束。

**怎么改**：在 schema 里加了**人工维护的「表间关联」列表**（如 A.id = B.a_id），在 prompt 里明确要求「多表必须且仅能使用该列表中的关联条件做 JOIN」，并在生成后做**校验**：多表 SQL 必须出现至少一条与配置关联一致的等式，否则拒绝并让模型重试。同时对写操作（DELETE/UPDATE）只生成不执行，需人工确认后再执行，避免误删误改。

**收获**：Text2SQL 多表不能完全交给模型自由发挥，要把「允许的关联」做成显式约束并写进 prompt + 后验校验；危险操作必须二次确认。

## 7. 响应变慢不知道卡在检索还是 LLM

**现象**：线上 P99 延迟明显升高，但不知道是检索慢、LLM 慢还是写库慢，只能凭感觉调。

**原因**：当时只有**端到端 latency_ms** 和 **route**（chat/qa/text2sql/rag），没有**分段耗时**。无法区分「RAG 请求里检索占多少、生成占多少」，只能根据 route 粗猜，调优缺乏依据。

**怎么改**：在观测表里保留了 route、latency_ms、RAG 的 retrieve_attempt、source_count、regenerate_count 等，用「RAG 且 retrieve_attempt 大、source_count 多」推断检索或重检耗时；用「regenerate_count 大」推断生成阶段多轮重试。同时文档里写了可扩展方向：在代码里对「缓存查询 / 总控 / 检索 / LLM 生成 / 持久化」等阶段分别打点，把 stage_ms 写入 trace 的 extra，这样就能直接看哪一阶段是瓶颈。目前先用现有字段做粗定位，后续再加分段耗时做细定位。

**收获**：可观测性要提前想好「出了问题要查什么」；至少要有请求级 ID、总耗时、路由和关键子步骤信息，再逐步加分段耗时，而不是事后才补。

## 8. 同一会话多端同时发消息，状态错乱或待确认 SQL 被覆盖

**现象**：用户在同一会话里开了两个标签页同时发问题，有时会出现「回答对不上问题」、或「点了确认执行却执行了别的 SQL」之类的情况。

**原因**：同一 thread_id 对应同一会话，但服务端没有对「同一会话」做并发控制。多个请求同时改图状态、写 pending_sql、写历史，就会出现交错和覆盖；确认执行时 pop 出来的 SQL 也可能不是用户当前看到的那条。

**怎么改**：做了**会话锁**：以 thread_id 为 key（按 hash 分桶控制锁数量），在进入「会修改该会话状态」的逻辑前加锁——包括 /chat、/chat/stream、确认执行、删除会话、重命名会话。同一会话的请求在这些接口上串行化，避免并发写导致状态错乱。

**收获**：有状态、有「当前待确认项」的会话型接口，要明确「同一会话的写是否允许并发」；不允许就要在入口用会话级锁串行化。
