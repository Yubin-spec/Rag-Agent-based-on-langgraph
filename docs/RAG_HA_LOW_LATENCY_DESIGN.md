# 高可用、低延迟 RAG 设计要点

本文从**高可用**和**低延迟**两个目标出发，整理 RAG 系统的设计思路，并对照本项目的实现做说明。面试或方案评审时可直接按此结构展开。

---

## 一、设计目标

| 目标 | 含义 | 典型手段 |
|------|------|----------|
| **高可用** | 依赖故障或抖动时，服务仍能响应、不雪崩、可恢复 | 熔断、降级、多节点、重试、健康检查 |
| **低延迟** | 首字/首包快、端到端时延可控 | 缓存、异步、连接复用、流式、检索与上下文裁剪 |

二者会互相影响：例如为保可用做重试会拉高延迟，为降延迟做缓存会引入一致性问题，需要在设计时做权衡。

---

## 二、高可用设计

### 2.1 依赖隔离与熔断

- **思路**：检索（向量库/BM25）、生成（LLM）、持久化（DB/Redis）任一不可用时，不把本服务拖垮。
- **做法**：
  - 对 DB、Milvus、LLM 分别做**熔断**：连续失败达阈值后短时不再请求，冷却后半开探测；避免下游慢或挂掉时连接/线程被占满形成雪崩。
  - **关键 vs 非关键**：必须成功的路径（如执行 Text2SQL）用 `critical=True`，失败抛错；对话历史、监控上报、RAG 检索结果等用 `critical=False`，失败时返回空/跳过并打日志，不阻塞主流程。
- **本项目**：`db_resilience.py` 统一管理 DB/Milvus 连接与熔断；`llm.py` 多 DeepSeek 节点按节点熔断并自动切换健康节点。详见 [高并发场景下的缓存、熔断与降级策略](HIGH_CONCURRENCY_CACHE_CIRCUIT_DEGRADATION.md)。

### 2.2 降级与可预期响应

- **思路**：依赖不可用或熔断时，不返回 500 或未定义行为，而是返回明确文案或空结果，便于前端/用户理解。
- **做法**：
  - 对话历史/监控写库失败 → 跳过写，对话照常进行。
  - RAG 向量检索失败（Milvus 熔断）→ 返回空 chunk，生成阶段可基于空上下文返回「未检索到相关内容」或建议重试。
  - LLM 调用多次重试仍失败 → 总控返回「当前服务暂时异常，请稍后重试或转人工」，不抛 500。
- **本项目**：总控 `supervisor_node_async` 在 LLM 重试耗尽后返回 `next=human` 并填入 `agent_need_human_reply`；RAG 检索与文档上传使用 `critical=False` + `default=[]` / `0`。

### 2.3 多节点与重试

- **思路**：单点故障时由其它节点接管；瞬时抖动用有限次重试消化。
- **做法**：
  - LLM：配置多个 endpoint，按健康状态与负载选节点；单节点可配置重试次数（如 2 次），多节点时每节点试一次，都失败再降级。
  - DB/向量库：获取连接或执行操作失败时指数退避重试（如 2 次），避免偶发网络/超时直接失败。
- **本项目**：`DEEPSEEK_API_ENDPOINTS` 多节点 + 熔断 + 按 `inflight/weight` 选节点；DB/Milvus 在 `safe_connection` / `milvus_operation_with_retry` 内重试。

### 2.4 健康检查与可观测

- **思路**：尽早发现依赖不可用，便于熔断生效和运维介入；关键路径有指标或日志。
- **做法**：提供 `/health`、`/health/ready`，探测 DB、Redis、Milvus、LLM 等；任一不可用时 ready 返回 503。熔断状态、降级次数、缓存命中率等可打日志或上报指标。
- **本项目**：健康检查接口存在；LLM 路由器提供 `get_deepseek_router_status()` 等状态快照，便于排查。

## 三、低延迟设计

### 3.1 多层缓存

- **思路**：能命中的请求尽量不走到最重的链路（检索 + LLM），缩短路径、降低延迟。
- **做法**：
  - **问答缓存**：问题归一化后作 key，命中则直接返回答案，不经过检索和 LLM。配合热 key 本地 LRU、防击穿单飞锁，高并发下仍稳、不击穿。
  - **检索缓存**：相同 query + 检索参数时复用「BM25 + 向量 + 重排」结果，只做一次 Milvus/BM25/重排，生成阶段仍走 LLM。适合重复问、相似问多的场景。
- **本项目**：`answer_cache.py` 负责问答缓存；`rag.py` 中 `_merge_3_7` 外层包进程内 LRU 检索缓存。详见 [Redis 问答缓存说明](REDIS_CACHE.md) 与 [高并发缓存、熔断与降级](HIGH_CONCURRENCY_CACHE_CIRCUIT_DEGRADATION.md#一缓存策略)。

### 3.2 异步与连接复用

- **思路**：IO 等待不占线程，连接复用减少建连开销。
- **做法**：
  - API 到图执行、LLM 调用全走异步（`ainvoke` / `astream`）；同步阻塞操作（DB、部分检索）用 `asyncio.to_thread` 放入线程池，避免阻塞事件循环。
  - DB、Redis、Milvus 使用连接池；按 URI/实例复用，避免每次请求建连关连。
- **本项目**：FastAPI 全 async 路由；LangGraph `ainvoke`/`aget_state`；DB 通过 `db_resilience` 连接池；Redis `redis.asyncio` 连接池。详见 [异步与高并发设计](ASYNC_CONCURRENCY.md)。

### 3.3 流式输出

- **思路**：用户感知延迟看「首字/首包时间」，流式可以边生成边返回，降低首包延迟、提升体验。
- **做法**：支持流式接口（如 SSE），检索完成后用 LLM `astream` 生成并逐 chunk 推送；前端收到首 chunk 即可开始展示。
- **本项目**：`POST /chat/stream`、知识库 `aquery_stream` 均支持流式；RAG 生成阶段使用 `_rag_chain` 流式输出。

### 3.4 检索与上下文控制

- **思路**：检索过多、上下文过长会拉高延迟和成本，需要在召回质量与延迟之间做平衡。
- **做法**：
  - 控制 `top_k`、重排后保留条数（如 `rerank_top`），避免一次拉过多 chunk。
  - 对注入 LLM 的上下文做长度/条数限制（如 `rag_max_context_chars`），超长则截断或摘要；单条消息也可设最大字符数，减少 token。
  - 检索质量评估 + 重检：匹配度过低时自动重检（如放宽条件或换策略），避免无效大上下文拖慢生成。
- **本项目**：`rag_min_match_score`、`rag_max_retrieve_attempts` 控制重检；`rag_max_context_chars` 限制注入上下文；RAG 混合检索 3:7 + 重排后取 top 若干条。

### 3.5 超时与限流

- **思路**：单请求不无限挂起；整体流量有上限，避免排队过长导致延迟恶化。
- **做法**：
  - LLM、DB、MinerU 等设置合理超时（如 `agent_request_timeout_seconds`），超时后触发重试或降级。
  - API 全局限流（如 Semaphore 限制同时处理请求数），超限排队或返回 503；对解析服务等也可做并发上限（如 MinerU 信号量）。
- **本项目**：`api_max_concurrent_requests`、`agent_request_timeout_seconds`、`mineru_concurrency_limit` 等；详见 [高并发缓存、熔断与降级](HIGH_CONCURRENCY_CACHE_CIRCUIT_DEGRADATION.md#四限流与并发控制)。

## 四、整体链路对照（本项目）

```
用户请求
  → 问答缓存命中？ → 是：直接返回（低延迟 + 无下游依赖）
  → 否：总控路由
       → 知识库：QA 精准匹配 → Text2SQL → RAG
                  → RAG：检索缓存命中？ → 是：复用检索结果，只走生成
                  → 否：Milvus（熔断则降级为空）+ BM25 + 重排 → 评估/重检 → 生成
       → 生成：LLM 多节点（熔断切换）+ 流式输出
  → 持久化/监控：非关键，失败降级跳过
```

高可用体现在：缓存/检索/LLM 任一环节故障或熔断时，有降级路径和明确响应；低延迟体现在：缓存、检索缓存、异步、流式、上下文与检索量控制。

## 五、面试时可用的概括话术

**「如何设计高可用、低延迟的 RAG？」**

- **高可用**：对 DB、向量库、LLM 做熔断和降级，区分关键路径和非关键路径；非关键（如历史持久化、监控）失败不阻塞主流程；LLM 多节点 + 重试，全部不可用时返回人工提示而非 500；配合健康检查和可观测。
- **低延迟**：问答缓存和检索缓存减少回源和重复计算；全链路异步和连接池减少等待；流式输出降低首包延迟；控制检索条数和上下文长度，并设超时与限流，避免单请求和整体排队过长。

需要展开时，可按「缓存 → 熔断/降级 → 多节点/重试 → 异步/流式 → 检索与上下文控制」逐点说，并带一句「我们项目里是在 XXX 模块通过 XXX 实现的」即可。
