# 如何设计上下文以提升模型推理能力

本文从**对话上下文**、**RAG 检索上下文**、**Prompt 结构**三方面说明如何设计「喂给模型的输入」，以提升推理效果与可控性；并对照本项目实现与可优化点。

---

## 一、设计原则（通用）

| 原则 | 说明 |
|------|------|
| **相关性** | 只送与当前任务强相关的内容，减少噪声与干扰，让模型把注意力放在关键信息上。 |
| **结构清晰** | 用编号、分段、标签（如 [证据1]、[背景]/[片段]、结论/依据）明确「哪段是依据、哪段是要求」，便于模型按结构推理与引用。 |
| **长度可控** | 在「信息充分」与「不超长」之间平衡；过长会占满上下文、稀释关键信息或触发截断，过短会丢失必要前提。 |
| **顺序与优先级** | 重要信息（如当前问题、核心约束）靠前或单独成段；历史与参考可「摘要在前、细节在后」或「最近 N 轮 + 旧摘要」。 |
| **角色与任务明确** | System 定角色与全局约束，Human 放本次输入与参考；避免把「要求」和「材料」混在一大段里。 |

## 二、对话上下文（多轮对话喂给总控/闲聊）

### 2.1 目标

总控要判断「闲聊 vs 知识库」，闲聊要结合历史做连贯回复；两者都需要**最近几轮 + 必要历史信息**，又不能无限制送全量历史（token 与噪音）。

### 2.2 本项目实现

| 手段 | 说明 |
|------|------|
| **滑动窗口** | 只取**最近 N 轮**（`llm_context_window_turns`，默认 10），每轮 = 1 条 user + 1 条 assistant；窗口内全量保留，不压缩。 |
| **旧对话摘要** | 若启用 `llm_context_summarize_old`，对**窗口外**的旧消息做一次 LLM 摘要（2～5 句话，保留诉求与结论），以一条 **SystemMessage「【历史对话摘要】+ 摘要正文」** 放在**最前**，再拼上最近 N 轮。这样模型既能看到「早期说了什么」，又不会堆满原文。 |
| **单条消息截断** | 对送入 LLM 的每条消息按字符数上限截断：**历史轮**用 `llm_context_max_chars_per_message_old`（如 600），**当前轮用户消息**用 `llm_context_max_chars_per_message_latest`（0 表示不截断）。SystemMessage（含摘要）不截断。避免某一条超长消息撑爆上下文。 |
| **摘要输入上限** | 做摘要时，送入摘要模型的对话文本受 `llm_context_summary_input_max_chars`（如 8000）限制，超长从末尾截断，控制摘要调用成本与质量。 |

**顺序**：`[历史摘要] + [最近 N 轮]`，每条再按需截断；当前轮用户消息在最后，模型优先看到「刚问什么」。

### 2.3 对推理的帮助

- **窗口**：保证模型看到的都是「近期相关」轮次，减少无关历史干扰。
- **摘要**：早期诉求与结论被压缩保留，总控/闲聊在做「是否知识库」「如何接话」时仍有依据，而不是只看到最近几句。
- **截断**：单条不爆长，上下文内 token 分布更均衡，关键句不被淹没。

## 三、RAG 检索上下文（喂给生成的参考内容）

### 3.1 目标

把检索到的 chunk 变成「模型易读、易引用、易遵守」的**结构化参考**，并约束输出格式与依据来源，从而提升**依据推理**与**可验证性**。

### 3.2 本项目实现

| 手段 | 说明 |
|------|------|
| **证据编号** | 每个 chunk 格式化为 `[证据i]\n[背景]\n父块\n[片段]\n子块`（无父块则仅 `[证据i]\n内容`）。模型在回答时被要求「事实后必须附 [证据N]」，便于溯源与事后校验。 |
| **背景 + 片段** | 父块提供段落背景，子块提供精确片段，模型既能理解「这段话在说什么语境」，又能精确引用片段，减少断章取义。 |
| **总长上限** | `rag_max_context_chars` 对拼接后的 context 做总字符上限，从前往后累加，超出则丢弃末尾 chunk，控制注入长度与 token。 |
| **场景化补充** | 按问题关键词从场景模板中选出 1～3 条「补充要求」（如「优先按流程顺序回答」「优先列出所需材料」），以 `scenario_guidance` 注入 human 段，引导模型**按业务场景组织答案**，而不是泛泛而答。 |
| **重试说明** | 当答案与参考关联度低或缺少引用时，将「上一版问题 + 请更严格贴合参考」写入 `retry_note`，再次调用时模型收到明确**反馈**，有利于第二轮推理纠偏。 |

**Prompt 结构**：System 定角色与约束（只依据参考、必须 [证据N]、结构化输出）；Human 依次放「参考内容」「用户问题」「场景化补充」「重试说明」，顺序固定，便于模型按块理解。

### 3.3 对推理的帮助

- **编号与结构**：模型有「可引用单元」，推理时更容易「先找证据再下结论」。
- **场景化**：缩小「该怎么答」的空间，模型更易按业务类型做结构化推理（流程、材料、条件等）。
- **重试说明**：把「哪里不对」显式告诉模型，相当于多了一轮**基于反馈的推理**。

## 四、Prompt 结构（System / Human 分工）

### 4.1 分工

- **System**：角色、任务、全局约束（禁止项、必须项、输出格式）。不随单轮变化，保证「每次调用同一套规则」。
- **Human**：本轮的**变量**——用户问题、检索 context、场景 guidance、重试说明等。结构清晰、分段明确，便于模型定位「问题在哪、依据在哪、要求在哪」。

### 4.2 本项目示例（RAG）

- System：海关政策助手；只依据参考、禁止编造；结构化输出（结论/依据/提示）；事实后必须 [证据N]；有重试说明时更严格贴合参考。
- Human：参考内容（带 [证据i]）→ 用户问题 → 场景化补充要求 → 重试说明（首轮为空）。

**对推理的帮助**：角色与约束稳定，模型不会「忘记」要引用证据；变量集中在 Human，模型只需在**固定结构**里填内容，减少自由发挥导致的违规。

## 五、可优化点（结合本项目）

| 方向 | 说明 |
|------|------|
| **对话上下文** | 可根据路由类型区分窗口：总控只需「最后一句 + 少量上文」即可路由，可考虑更小窗口或只送最后 1～2 轮；闲聊可保持或略大。减少总控 token 与延迟。 |
| **摘要质量** | 摘要 prompt 可强调「保留与后续决策相关的信息」（如用户明确说过的业务词、已做结论），或按轮次加权（越近的轮次摘要里占比略多）。 |
| **RAG 上下文顺序** | 检索结果可按与问题的**相关度排序**后再拼 context，把最相关的 chunk 放在前面，提升模型「先看到最重要依据」的概率。 |
| **分阶段注入** | 若 context 极长，可考虑「先送问题 + 最相关 2～3 条，模型若请求更多再送下一批」，需改造成本；当前用总长上限已能控制。 |
| **推理链显式化** | 若希望模型「先列依据再得结论」，可在 prompt 中要求输出格式为「依据：…；结论：…」，或引入 CoT 风格步骤（本项目当前为直接结论+依据）。 |
| **负面示例** | 在 System 或 Human 中加 1～2 句「错误示例：不要…」，减少常见违规（如无证据表述、泛泛而谈）。 |

---

## 六、小结（面试可答）

- **设计原则**：相关性、结构清晰、长度可控、顺序与优先级、角色与任务明确。
- **对话上下文**：滑动窗口（最近 N 轮）+ 旧对话摘要（窗口外压缩为一条）+ 单条消息字符截断；顺序为「摘要 + 最近 N 轮」，当前轮在最后。
- **RAG 上下文**：证据编号（[证据i]+背景+片段）、总长上限、场景化 guidance、重试说明；Prompt 中 System 定约束，Human 放参考/问题/补充/反馈。
- **对推理的提升**：结构化和编号便于引用与校验；摘要保留早期信息；场景化与重试说明引导模型按业务推理并纠偏；Prompt 分工清晰减少违规。优化可做在窗口分层、摘要质量、检索排序、分阶段注入与显式推理格式等。
