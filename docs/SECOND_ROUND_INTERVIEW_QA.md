# 二面高频问题与参考答案

适用场景：上海算法负责人二面，围绕海关知识库项目，重点考察项目真实性、技术原理、方案取舍、扩展思路、协作沟通与复盘能力。

使用建议：
- 先通读一遍，形成统一口径。
- 每题不要死背，重点记住回答结构。
- 数字类问题如果不确定，不要硬报，用“基于业务样例和人工验收调优”替代。

## 一、项目整体与架构

### 1. 你先整体介绍一下这个项目
**参考答案：**

这是一个面向海关业务场景的智能问答系统，目标是解决知识分散、查询效率低、结构化和非结构化数据难统一使用的问题。整体上我把它拆成四层：接口层用 `FastAPI` 提供聊天、流式输出、文档上传、Text2SQL 配置审核和监控反馈；编排层用 `LangGraph Supervisor` 做路由；知识问答层按 `FAQ -> Text2SQL -> RAG` 依次尝试；基础设施层用 `Milvus` 做向量检索、`Redis` 做缓存、`PostgreSQL` 做长期记忆和问答监控。我的核心工作主要在知识问答主链路设计，以及 RAG、Text2SQL 和工程化落地。

### 2. 为什么要做成 FAQ、Text2SQL、RAG 三段式？
**参考答案：**

因为不同问题的最优解不一样。固定高频问答最适合 FAQ，成本低、响应快、结果最稳定；结构化统计类问题更适合 Text2SQL，因为数据库才是最可靠的数据源；复杂政策解读、流程说明、跨文档归纳类问题才适合 RAG。所以我没有把所有问题都丢给一个链路，而是让不同类型的问题走更适合自己的路径，这样准确率和成本都更可控。

### 3. 你在这个项目里的核心贡献是什么？
**参考答案：**

如果只总结三点，我会聚焦：第一，设计了知识问答主链路，也就是 `FAQ -> Text2SQL -> RAG` 的分层方案；第二，把 RAG 从“能跑”做到了“有证据、可校验、尽量低幻觉”；第三，把系统工程化，包括缓存、流式输出、长期记忆、人工确认执行、问答监控和反馈闭环。我的价值不只是把组件串起来，而是把它做成一个能落地、能维护、能观察效果的业务系统。

### 4. 这个项目更偏算法项目还是工程项目？
**参考答案：**

我觉得它是一个 AI 应用工程化项目，不是做底层模型训练，也不是纯 CRUD。它既需要工程能力把链路做完整，又需要对检索、重排、grounding、schema 语义和路由这些核心模块有算法理解。对我来说，它更像“工程驱动、算法增强”的 AI 系统落地项目。

### 5. 为什么用多智能体，而不是一个大 Prompt 全做？
**参考答案：**

因为不同问题类型的处理方式差异很大。闲聊、知识问答、高风险操作、人工介入的目标不同，如果都塞进一个大 Prompt，系统会变得不稳定、难维护，也很难做状态控制。多智能体或者更准确地说是多节点编排的价值，在于把不同职责拆开，让路由和异常兜底更清晰，也方便后续扩展。

### 6. 你们这个项目的业务价值体现在哪里？
**参考答案：**

它的业务价值主要有三点：第一，把海关内部散落的知识和数据查询能力统一成自然语言入口，降低使用门槛；第二，通过 FAQ、Text2SQL、RAG 分流，让不同问题得到更合适的处理方式；第三，系统不是一次性 demo，而是有会话管理、反馈闭环和监控分析能力，方便后续持续优化。

### 7. 这个项目最难的点是什么？
**参考答案：**

我觉得最难的不是某一个单点模块，而是把多个能力拼成一个长期可用的系统。比如 Text2SQL 难在 schema 语义的长期维护，RAG 难在检索质量和幻觉控制，工程上还要处理缓存、状态恢复、人工确认和监控反馈。这些问题单独看都不算新，但要一起做好并长期稳定运行，难度就上来了。

### 8. 如果让你现在重新做一版，你会重点改哪三点？
**参考答案：**

第一，我会把评估体系做得更标准化，把检索、生成、用户反馈分层评估；第二，我会把 Text2SQL 的 schema 语义维护做得更后台化，减少对开发介入的依赖；第三，我会进一步增强复杂 query 的检索能力，比如 query decomposition 或 iterative retrieval，而不是只停留在单轮 RAG。

### 9. 你怎么证明这个项目不是简单拼装组件？
**参考答案：**

我会从两个角度回答。第一，主链路不是简单堆组件，而是根据问题类型做路由和取舍，比如 FAQ、Text2SQL、RAG 三种方案的边界很不一样；第二，落地过程中真正费力的是优化和控制，比如检索权重、`topk`、上下文管理、回答 grounding 校验、SQL 安全约束、长期记忆和反馈闭环，这些都不是把库装上就自动有的。

### 10. 你觉得这个项目最大的边界是什么？
**参考答案：**

最大的边界有两个。一个是复杂 query 下，单轮 RAG 的能力是有限的，如果问题本身需要多跳证据链，后续需要引入更强的检索规划能力；另一个是 Text2SQL 虽然能自动同步 schema，但新增字段的业务语义理解仍然依赖人工维护，这决定了它的长期效果上限。

## 二、RAG 设计与原理

### 11. 你讲一下你们项目里的 RAG 是怎么做的
**参考答案：**

我们做的是一个以单轮检索为主的业务型 RAG，重点不是把链路做得特别复杂，而是把检索质量和回答可靠性做稳。文档侧先做解析和父子切片，子块负责检索精度，父块负责补语义背景；检索侧用混合召回，向量检索为主，全文检索做补充，再用 reranker 做二次排序；生成时把命中的证据组织成结构化上下文给模型；生成后再做 grounding 校验，如果答案和证据相关度不够，就重生成或者保守拒答。

### 12. 为什么要用父子切片？
**参考答案：**

因为海关文档很多是条款、流程、条件、附注这类强结构化内容，如果只用固定长度切片，容易命中局部信息但缺少约束上下文。父子切片的作用是，子块提高命中精度，父块补足语义背景。这样在检索和生成之间做了一次平衡，既不让块太大影响召回，又尽量避免断章取义。

### 13. 你们为什么要做混合召回，而不是只用向量检索？
**参考答案：**

因为向量检索和全文检索各有优势。向量检索更擅长语义召回，但对法规编号、专业术语、固定表达未必稳定；全文检索更适合精确关键词命中，尤其在海关这种专业词很多的场景下能提供补充。所以我更倾向于把两者组合起来，再通过重排统一排序，而不是把它们看成二选一。

### 14. reranker 是怎么筛噪声的？
**参考答案：**

初召回阶段更偏 recall，会混入一些相关但不够直接回答问题的候选。reranker 的作用就是对 query 和每个候选片段做更细粒度匹配，把真正更能回答当前问题的证据排前面，把关键词命中但其实偏背景、偏弱相关的片段往后压。后续只把前几条高分候选给大模型，相当于在生成前做了一次去噪。

### 15. topk 为什么不能一直调大？
**参考答案：**

`topk` 太小会漏召回，关键证据可能根本进不了候选集；但 `topk` 太大也会引入大量噪声，增加 reranker 负担，并污染最终给模型的上下文。RAG 里最怕的不是没有内容，而是给模型太多半相关内容，让它乱综合。所以调 `topk` 的目标不是越大越好，而是让最终证据既够用又干净。

### 16. 你们是怎么调 RAG 效果的？
**参考答案：**

主要围绕四层调优。第一层是切片策略，比如 chunk size、overlap、父子块配合；第二层是召回策略，比如全文和向量的权重、初召回 `topk`；第三层是排序和上下文组织，比如 rerank 后保留多少、最终上下文怎么拼；第四层是生成和回答后校验，比如证据引用、grounding 阈值、低相关时是否重生成。也就是说，我不是只调一个参数，而是从召回、排序、生成三个阶段一起看。

### 17. 你们怎么判断 RAG 是否出现幻觉？
**参考答案：**

我更倾向于把幻觉理解成“答案是否真正被证据支持”。所以我们生成完答案后，会再把答案和本次检索到的证据做一轮 grounding 校验。如果答案和证据的相关度太低，或者答案里没有证据编号，就认为存在较高风险，这时要么重生成，要么直接拒答，而不是把一个看起来流畅但可能错误的答案直接返回给用户。

### 18. 重生成真的能解决幻觉吗？
**参考答案：**

不能说一定解决，它更像是最后一层风险缓释。如果底层检索证据本身就不够，单纯重生成意义不大，最好的做法是拒答或让用户补充问题。所以我不会把重生成当成万能方案。它更适合处理“证据基本够，但模型第一次回答跑偏”的情况。

### 19. 什么时候单轮 RAG 不够？
**参考答案：**

当问题本身具有明显多跳特征时，单轮 RAG 往往不够。比如涉及政策更新前后对比、跨文档依赖、多个条件链路拼接、先找文件再找条款这类问题，第一轮检索可能只能找到线索而不是最终答案。这个时候就更适合用 query decomposition 或 iterative retrieval 去增强。

### 20. query decomposition 是什么？
**参考答案：**

它就是先把一个复杂 query 拆成多个更清晰的子问题，再分别检索和聚合结果。适合原问题过长、条件太多、一个 query 同时包含多个维度的场景。它的本质是先拆问题，再检索，让每个子任务更明确。

### 21. iterative retrieval 是什么？
**参考答案：**

它是指检索不是一次完成，而是先检索一轮，判断当前证据够不够；如果不够，就根据当前高置信证据或信息缺口继续检索下一轮。它比单轮 RAG 更适合多跳问题，但前提是第一轮证据质量足够高，否则会把误差继续放大。

### 22. 如果第一轮检索不准，iterative retrieval 会不会更糟？
**参考答案：**

会，所以 iterative retrieval 不能盲目基于第一篇文档往下搜。更合理的做法是先对第一轮高分候选做置信度判断，只从高质量证据里提取下一轮检索线索；如果第一轮信号太弱，我会更倾向于重写 query 或重新召回，而不是强行多跳。

### 23. 你们怎么评估 RAG 效果？
**参考答案：**

我会把评估拆成三层。第一层是检索层，看召回相关性、命中位置、噪声比例；第二层是生成层，看答案是否真正基于证据，是否带证据引用；第三层是业务层，看用户问题是否被解决。当前项目里更多是基于业务样例和人工验收调优，如果继续做，我会把 query-doc-answer 三层评估做得更标准化。

### 24. 你们为什么不直接做 agentic RAG？
**参考答案：**

因为当时业务目标是先把单轮链路做稳，而不是一开始就把系统复杂度拉得很高。agentic RAG 或复杂多跳检索当然更强，但同时也引入了更高的状态管理、评估和误差传播问题。在当前项目阶段，我更关注检索质量和回答可靠性是否足够可控。后续如果问题复杂度持续提升，再考虑引入更强的检索规划能力。

### 25. 你觉得 RAG 项目真正拉开差距的地方是什么？
**参考答案：**

我觉得不是“有没有接上向量库”，而是三件事：检索质量、证据约束、回答可靠性。基础流水线大家都差不多，真正拉开差距的是你能不能稳定召回对的问题证据，能不能让模型尽量只基于证据回答，以及能不能在回答后再做一层可靠性控制。

## 三、Text2SQL 设计与优化

### 26. 你讲一下你们项目里的 Text2SQL
**参考答案：**

Text2SQL 主要是解决结构化查询问题，比如数量统计、条件筛选、名单查询、按时间范围统计等。这类问题最可靠的数据源是数据库本身，不适合用 RAG 去文档里猜。我们的 Text2SQL 不是让模型自由写 SQL，而是一条受控生成链路：先读取 schema 和人工维护的业务语义给模型，再生成 SQL，然后做只读约束、语法校验、多表关系校验，最后执行并把结果转成自然语言答复。

### 27. schema 是怎么给到大模型的？
**参考答案：**

不是直接把数据库原始 metadata 原样塞给模型，而是先整理成 LLM 友好的 schema block。里面包括表名、字段名、字段类型、人工维护的表含义、字段含义以及表间关联。最终它会作为 Text2SQL system prompt 的一部分注入给模型。也就是说，模型看到的不只是表结构，还能看到一些业务语义和 join 规则。

### 28. schema 为什么不能只给表名和字段名？
**参考答案：**

因为很多字段名本身并不能说明业务含义，比如 `status`、`type`、`flag`、`code` 这类字段，模型就算看到了，也未必知道它在海关场景里具体代表什么。Text2SQL 很多错误本质上不是 SQL 语法问题，而是业务语义映射错了。所以必须补表含义、字段含义和表间关系。

### 29. Text2SQL 为什么不能完全靠 prompt？
**参考答案：**

因为它不只是文本生成问题，还涉及执行安全和语义约束。如果只靠 prompt，模型很容易选错表、乱 join、生成危险 SQL，或者在字段名相似时映射错误。所以我更倾向于把 Text2SQL 做成“模型生成 + 规则约束 + 执行闭环”的方案，而不是纯 prompt 驱动。

### 30. 你们怎么避免模型乱 JOIN？
**参考答案：**

多表查询时，我们不会让模型自己随意猜关系，而是要求它必须使用人工维护的表间关联。换句话说，大模型能生成 SQL，但 join 条件的合法空间是被规则收窄的。生成后还会额外校验是否真的使用了这些关系，避免出现语义上看着对、实际上是笛卡尔积的 SQL。

### 31. 只允许 SELECT 的原因是什么？
**参考答案：**

因为自动执行路径必须优先保证安全。查询型问题最常见，也最适合自动化；而删除、修改、插入类操作会带来明显风险，不应该直接交给模型自动执行。所以自动路径只允许 `SELECT`，写操作最多只生成 SQL，不自动落库，必须走人工确认。

### 32. 如果用户真的要删数据，你们怎么处理？
**参考答案：**

系统会识别删除或修改意图，但不会直接执行，而是生成对应的 `DELETE` 或 `UPDATE` SQL，并要求人工确认。也就是说，大模型负责帮助生成候选 SQL，但最终执行权限在人工确认环节。这是典型的高风险操作兜底机制。

### 33. Text2SQL 准确率不够时你怎么优化？
**参考答案：**

我会分层处理。第一层先补 schema 语义，因为很多错误本质上是模型不理解字段业务含义；第二层做 schema pruning，只把当前 query 相关的表送给模型，减少选错表和字段的概率；第三层加强约束，比如只读、关系限定、语法校验；第四层做执行闭环，把错误信息反馈给模型修正；第五层再看 query 本身是否模糊，必要时做 query rewrite 或澄清。

### 34. Text2SQL 最大的难点是什么？
**参考答案：**

我觉得最大的难点不是生成 SQL 本身，而是 schema 语义的长期维护。数据库表结构变了，技术上可以重新读取，但新增字段的业务含义模型不一定理解。项目期内可以人工补说明，但从长期来看，真正难的是如何让这种语义维护不依赖开发改代码，而是能够后台化、可持续维护。

### 35. 如果 schema 频繁变化怎么办？
**参考答案：**

技术上可以通过缓存加定时刷新去同步最新 schema，这个问题相对好解决。更难的是 schema 变更之后，业务语义是否也同步被维护了。所以我会把问题分两层：底层结构自动同步，业务语义通过人工审核配置持续维护。只有这两层都在，Text2SQL 才会长期稳定。

### 36. 为什么不直接做模板 SQL？
**参考答案：**

如果问题类型特别固定，模板当然更稳，但扩展性会比较差。我们这个场景里问法差异较大，而且查询条件组合很多，如果全靠模板，维护成本会迅速变高。所以我的做法是让模型负责自然语言到 SQL 的映射，再通过规则和校验把它收在安全边界内，这样兼顾灵活性和可控性。

### 37. 你们怎么处理“没查到数据”这种情况？
**参考答案：**

如果 SQL 执行成功但结果为空，不会强行让模型编造，而是明确返回“未查询到符合条件的数据”。因为对用户来说，空结果本身也是有效信息。相比之下，编造一个看似合理但实际上不存在的结果，风险要大得多。

### 38. Text2SQL 结果返回后为什么还要再经过一次大模型？
**参考答案：**

因为数据库结果对业务用户来说通常不够友好，尤其是多列、多行结果集。再次经过 LLM 的目的，不是让模型再“创造内容”，而是把结构化查询结果转成更自然的业务答复。但这一步前提是必须基于真实结果生成，不能脱离结果去发挥。

### 39. 如果让你把 Text2SQL 再往深做，你会做什么？
**参考答案：**

我会考虑两件事。第一是把表级和字段级候选召回做得更好，也就是先解决“当前 query 应该看哪些表和字段”；第二是让错误反馈更结构化，比如 execution-guided 修正。这样可以把 Text2SQL 从“靠大模型猜”进一步升级成“在小范围、高约束下生成”。

### 40. 你怎么定义一个好的 Text2SQL 系统？
**参考答案：**

我不会只看 SQL 生成率，而会看三件事：一是 schema 语义是否足够清晰；二是 SQL 是否安全、可执行、口径正确；三是用户最终是否拿到了正确且可理解的业务答案。真正好的 Text2SQL，不只是“模型会写 SQL”，而是“自然语言查询能被安全、可控地映射为数据库能力”。

## 四、工程化、性能与可维护性

### 41. 你们为什么做缓存？
**参考答案：**

因为业务系统里会有不少高频重复问题，如果每次都重新走完整条链路，会浪费推理资源，也增加延迟。所以我们用缓存去承接那些答案稳定、时效性不强的问题。它的价值不只是快，更重要的是降低重复推理成本。当然缓存不是对所有问题都适合，像强实时查询和强上下文问题就不能简单复用。

### 42. 为什么要做流式输出？
**参考答案：**

流式输出的价值不只是“像 ChatGPT”，而是改善首字节体验。对于用户来说，主观等待感受往往比总耗时更敏感。尤其在 RAG 或长答案场景下，哪怕总时长差不多，只要能更早看到系统开始回应，体验就会明显好很多。

### 43. 你们为什么要做长期记忆？
**参考答案：**

短期会话状态放在内存里虽然快，但进程重启后会丢失，也不适合更长周期的会话恢复。所以我们加了 PostgreSQL 去保存对话历史和运行时状态，比如待确认 SQL、人工中断状态等。这样同一个会话即使进程重启，也能恢复上下文，系统更像一个真正可持续使用的业务应用。

### 44. 长对话为什么不能把所有历史都直接塞给模型？
**参考答案：**

因为这样会带来两类问题：一是 token 成本快速上升，二是上下文噪声越来越多。所以更合理的做法是保留最近几轮原文，把更早的历史压缩成摘要，再一起喂给模型。这样兼顾成本和上下文完整性，也更符合真实系统的工程约束。

### 45. 你们做问答监控的目的是什么？
**参考答案：**

监控的目的不是为了“记录日志”，而是为了形成可迭代闭环。系统不仅要知道这次问题走了哪条链路、耗时多少、命中了哪些证据，还要知道用户最终觉得有没有帮助、在哪些场景下容易出问题。只有这样，后续才能基于真实数据去优化 RAG、Text2SQL 和路由策略。

### 45.1 如果这个项目效果出现问题或效果不理想，如何快速定位？
**参考答案：**

我会按「先定环节、再定原因」来答。第一，先区分是**异常挂了**还是**慢或质量差**：异常看日志和健康检查——各模块都有专门的异常处理和日志等级（见 `docs/LOGGING.md`），能看出是 DB、Redis、Milvus 还是 LLM 出了问题；`/health` 会探所有依赖并返回熔断状态，具体到达哪个节点可以看 LangSmith。第二，**请求级**：每次问答有 `observation_id`，会落库到观测表，包含 route、耗时、质量标签、RAG 的 grounding_score、检索命中的 chunk_id 等；RAG 还有单独的 trace 表存 final_status、retrieve_attempt、top_match_score。第三，**质量与瓶颈**：用 `/qa/analytics` 看差评案例、反馈标签和场景统计，按 route 和 quality_label 能快速看出是检索没命中、生成没对上文档，还是超时/熔断；结合 trace 里的 grounding_score 和 regenerate_count 可以判断是检索阶段还是生成阶段的问题。所以总结就是：日志和 health 定是否挂、在哪个依赖；观测表和分析接口定是哪个环节（路由/检索/生成）；需要更细的节点和耗时就看 LangSmith。详细排查路径和表格见 `docs/EFFECT_DEBUGGING_AND_LOCALIZATION.md`。

## 五、协作沟通与复盘

### 46. 你在项目里是怎么和产品或业务侧沟通的？
**参考答案：**

我一般会先把需求拆成三层：业务目标、数据条件、技术边界。先确认到底要解决什么问题，再判断更适合走 FAQ、Text2SQL 还是 RAG。如果我觉得需求里有模糊点或者风险，我不会直接说做不了，而是会把风险、成本和替代方案讲清楚，比如先做一个可控版本，再逐步扩展复杂能力。这样能减少后面反复返工。

### 47. 如果产品方技术理解不够，你怎么处理？
**参考答案：**

我会尽量避免站在“你懂不懂技术”的对立面，而是把问题翻译成更容易共识的方式。比如不直接讲“这个方案技术上很难”，而是说“如果按这个做，风险在哪里、效果可能不稳定、交付时间会怎样变化，同时我可以提供哪个替代方案”。我更倾向于把协作做成边界对齐，而不是谁说服谁。

### 48. 如果技术方案和产品诉求冲突，你怎么推进？
**参考答案：**

我的做法通常是分版本处理。先明确短期最重要的业务目标，看看能不能先上线一个更可控、更稳定的版本，再把复杂能力放到后续迭代里。这样既不会因为追求完美拖慢交付，也不会为了赶进度直接埋下很大的技术债。很多冲突本质上不是谁对谁错，而是优先级排序问题。

### 49. 你怎么看独立推进和团队协作的关系？
**参考答案：**

我能接受独立推进，也愿意先自己把问题想透、做出雏形，但我不觉得独立就等于闭门做事。真正高效的方式还是在关键节点及时对齐，比如方案评审、数据口径确认、风险暴露和上线复盘。我比较认可的是“先主动探索，遇到关键分歧及时沟通”，而不是一开始就全依赖别人或者完全不对齐。

### 50. 如果面试官最后让你总结这个项目的成长收获，你怎么说？
**参考答案：**

这个项目让我最大的收获，是我不再把 AI 应用理解成单纯调模型，而是要把问题类型、数据来源、检索质量、执行安全、用户体验和长期维护一起考虑。它让我更明确自己适合做的方向，是 AI 应用系统落地和工程化，不只是把模型接上，而是把它做成一个真正可用、可控、可迭代的业务系统。

## 六、临场回答原则

### 1. 数字不确定时怎么说
- 不要硬报准确率、比例、阈值。
- 可替换成：“当时主要基于业务样例和人工验收调优。”

### 2. 答案结构建议
- 先讲业务问题
- 再讲设计取舍
- 再讲具体实现
- 最后讲边界和后续优化

### 3. 常用稳妥句式
- “我们当时先把单轮可控链路做稳，没有一开始把系统做得特别复杂。”
- “这个点确实是当时方案的边界，如果继续优化，我会优先从 A 和 B 两个方向做。”
- “我更关注的是业务可用性和系统可控性，而不是单纯把链路堆复杂。”

## 七、算法负责人压力追问版 20 题

### 51. 你这个 RAG 听起来就是标准流水线，技术含量体现在哪里？
**参考答案：**

如果只看主链路，确实是标准单轮 RAG，不是特别复杂的 agentic 或多跳系统。但我的工作重点不是堆复杂度，而是把检索质量和回答可靠性做稳。比如切片粒度、混合召回、`topk`、rerank、上下文组织、grounding 校验，这些都会直接影响最终效果。对业务系统来说，真正有价值的不是链路名字多复杂，而是它能不能稳定给出可信答案。

### 52. 既然是标准 RAG，为什么不直接买现成方案？
**参考答案：**

现成方案可以快速起步，但海关这个场景对结构化查询、证据约束、人工确认执行和会话工程化都有比较强的定制需求。比如 FAQ、Text2SQL、RAG 三段式分流，Text2SQL 的关系约束和人工确认，以及回答后的 grounding 校验，这些都不是通用产品默认能很好覆盖的。所以我们不是单纯为了“自己造”，而是因为业务边界决定了需要更深定制。

### 53. 你们为什么不直接上更强模型，而要做这么多检索和校验？
**参考答案：**

更强模型当然可能提升上限，但它不能替代数据来源可靠性和系统可控性。尤其在海关这种高风险业务场景里，问题不是“模型能不能说”，而是“答案是否真的有证据支持，能不能解释，能不能控制风险”。所以我更倾向于把模型能力和检索、证据约束、执行安全结合起来，而不是把所有问题都寄托在模型本身更强上。

### 54. 为什么 FAQ 不直接并进 RAG，一起用向量检索解决？
**参考答案：**

因为 FAQ 场景的最优解通常不是语义检索，而是精准命中。固定高频问答如果已经有标准答案，直接匹配会更快、更稳，也更省资源。把它们全部并进 RAG，反而会增加不必要的召回和生成成本，也会让结果稳定性下降。所以 FAQ 单独作为一层，本质上是成本和稳定性的取舍。

### 55. 你怎么证明 reranker 真的有效，而不是多此一举？
**参考答案：**

我会看两个现象。一个是没有 reranker 时，初召回里经常会混入一些相关但不直接回答问题的片段；另一个是加上 reranker 后，最终给模型的证据更集中，回答会更聚焦，grounding 也更稳定。它的价值不是“让没召回到的内容凭空出现”，而是在候选集已有相关内容的前提下，把真正更关键的证据提到前面，减少噪声上下文。

### 56. 如果向量检索已经很准了，为什么还需要全文检索？
**参考答案：**

向量检索擅长语义相近，但对法规编号、专有术语、固定表达未必最稳。海关场景里恰好这类内容很多，所以全文检索能作为很好的补充。我的理解不是全文检索替代向量检索，而是用它补齐向量检索在精确术语场景下的短板。

### 57. 你说 topk 要调，那怎么避免它变成拍脑袋调参？
**参考答案：**

我不会只看单个参数，而是看调参前后出现的系统现象。比如是否经常漏掉关键证据、rerank 后前几条是否稳定、最终上下文噪声是否增加、回答是否更散、grounding 是否下降。如果这些现象有明确变化，就说明 `topk` 调整带来了系统性影响。也就是说，`topk` 不是孤立调，而是和召回质量、上下文纯度、回答稳定性一起看。

### 58. 你们为什么不做 query rewrite？
**参考答案：**

不是完全不能做，而是当时项目阶段更优先先把基础链路做稳，比如切片、召回、rerank 和回答后校验。query rewrite 会增加一层链路复杂度和评估成本，如果在底层召回质量还没跑稳时加太多增强策略，反而难以定位问题来源。后续如果复杂 query 比例更高，我会考虑把 query rewrite 作为针对性增强手段加进去。

### 59. 你知道 multi-hop retrieval，但你们为什么没做？
**参考答案：**

因为 multi-hop retrieval 更适合本身就需要证据链的问题，而不是所有问题都要多轮检索。我们当前场景里，大部分问题还是单轮可解，先把单轮链路的检索质量和回答可靠性做稳，投入产出比更高。多跳检索当然是后续可扩展方向，但它的前提是第一轮证据质量足够高，否则会放大误差。

### 60. 你们的 grounding 校验会不会把正确答案也误杀掉？
**参考答案：**

有这个风险，所以 grounding 校验不能设计成“一刀切”，而应该作为风险控制机制，而不是绝对裁判。它更适合识别明显脱离证据的答案，而不是代替所有正确性判断。实际落地时，我会把它理解成一个保守过滤器，低分时优先触发重生成或兜底拒答，而不是简单粗暴地判定答案一定错。

### 61. Text2SQL 为什么不直接用数据库注释，为什么还要人工维护 schema 语义？
**参考答案：**

因为数据库注释并不总是完整、规范或者面向业务的。很多真实系统里，字段名和注释只能说明技术含义，不能完全表达业务口径。尤其是多部门协同的数据表，模型最容易错的不是 SQL 语法，而是字段背后的业务语义。所以人工维护 schema 语义不是重复劳动，而是把“数据库结构”转成“模型可理解的业务上下文”。

### 62. 既然 schema 语义这么重要，为什么不做成知识图谱？
**参考答案：**

知识图谱当然可以更系统地表达实体和关系，但建设成本和维护成本都更高。对于我们当时的项目阶段来说，先把表含义、字段含义和表间关系做成轻量可维护配置，投入产出比更高。换句话说，不是图谱不行，而是要看当前业务规模、人力和维护能力是否匹配。

### 63. Text2SQL 最难的是生成 SQL，还是别的？
**参考答案：**

我认为最难的不是语法生成本身，而是 schema 语义理解和长期维护。SQL 语法错误可以靠约束、校验和重试去修正，但如果模型对字段含义理解错了，它可能会生成一条语法完全正确、但业务语义完全不对的 SQL。这个问题比语法问题更隐蔽，也更危险。

### 64. 为什么写操作不直接禁掉，还要允许生成候选 SQL？
**参考答案：**

因为有些业务场景下，用户确实会有删除、修改的需求。如果完全禁掉，系统的能力边界就太死了。允许生成候选 SQL 的意义是让系统具备辅助能力，但不把最终执行权交给模型。也就是说，模型帮助理解用户意图并生成候选方案，真正的高风险动作仍然由人工确认，这样能兼顾能力和安全。

### 65. 如果模型生成 SQL 总是选错表，你先改什么？
**参考答案：**

我会先从 schema pruning 和 schema 语义增强入手，而不是先换模型。因为选错表很多时候不是模型不会推理，而是面对全库 schema 时搜索空间太大、字段语义又不够清晰。如果能先把候选表缩到少量高相关表，再补充字段含义和关系，准确率通常会有更直接的提升。

### 66. 你们为什么不做一个专门的 intent classifier 来路由问题？
**参考答案：**

从长期看，专门的 classifier 当然是可选方向，但项目早期我更倾向于用更轻量、可快速迭代的方案，比如启发式触发和规则路由。因为当时首先要验证的是整体链路是否跑得通，以及 FAQ、Text2SQL、RAG 的边界是否清楚。后续如果积累了更多真实 query 数据，我会更愿意把路由升级成分类模型。

### 67. 你说自己做了很多优化，那你怎么判断某次优化真有帮助？
**参考答案：**

我会看三个层面。第一是模块层，比如召回结果是否更聚焦、SQL 执行成功率是否更高；第二是回答层，比如 grounding 是否更稳定、空答和错答是否减少；第三是业务层，比如人工验收和用户反馈是否更好。只有这些层面形成一致趋势，我才会认为这次优化是有效的，而不是偶然个例。

### 68. 你们项目里有没有什么设计是现在回头看会推翻的？
**参考答案：**

如果说推翻谈不上，但我会认为当时对评估体系的建设还不够前置。很多优化是在业务样例和人工观察基础上做的，虽然能推动系统变好，但如果更早把检索、生成、路由的评估体系做起来，后面的很多取舍会更有数据支撑。这是我现在回头看最想补的一块。

### 69. 如果让你带一个新人接手这个项目，你会最先教他什么？
**参考答案：**

我会先让他理解问题分层，而不是先看代码。先明白为什么有 FAQ、Text2SQL、RAG 三条链路，各自解决什么问题；再理解 RAG 的重点不只是召回，而是证据质量和回答可靠性；最后再看工程化部分，比如缓存、状态、监控和反馈。因为这个项目如果只从代码细节看，很容易把它误解成组件拼装，真正的关键其实在设计边界和取舍。

### 70. 如果我们这里的项目和你之前的不一样，你怎么保证自己也能快速上手？
**参考答案：**

我对自己的判断不是“我只会做海关项目”，而是我已经经历过一类典型 AI 应用系统落地：如何拆问题、如何选数据源、如何做受控生成、如何平衡效果和风险、如何做反馈闭环。具体业务会变，但这些方法论是可以迁移的。所以我会更有信心在新场景里先快速理解问题边界，再把合适的能力组合起来，而不是机械照搬上一个项目。

## 八、面试结束时可反问的 8 个问题

使用建议：
- 不要一次性全问，挑 2 到 4 个最合适的问。
- 优先问和团队、项目、技术路线相关的，不要上来先问福利。
- 反问的目标不是“凑流程”，而是体现你在认真判断岗位匹配度。

### 1. 如果我加入，前 3 个月最希望我解决的核心问题是什么？
**适用原因：**

这个问题能体现你关注的是落地和价值，而不是只关心 title。也能帮助你判断这个岗位当前最急的方向，是 RAG、Agent、推荐系统，还是某个具体业务线。

### 2. 目前团队做 AI 应用时，最大的技术挑战更偏效果、工程，还是产品落地？
**适用原因：**

这个问题很适合问算法负责人，因为他通常能从更高视角看团队当前的主要矛盾。你也能借此判断自己更适不适合这个团队。

### 3. 团队现在在 RAG、Agent、Text2SQL 或知识库方向上，最近最关注的演进点是什么？
**适用原因：**

这个问题既能体现你对技术方向有持续关注，也方便你判断团队做的是基础版应用，还是已经在往更复杂的检索规划、多智能体协同方向演进。

### 4. 你们更希望这个岗位的人，偏快速落地型，还是偏方法研究和效果优化型？
**适用原因：**

这个问题很关键，因为很多岗位 JD 写得很像，但实际要求可能差很多。你可以借此判断他们更看重工程推进，还是更看重算法深挖。

### 5. 目前团队内部，算法、后端、产品之间是怎么协作的？技术方案一般是如何定下来的？
**适用原因：**

这个问题能帮助你判断团队协作方式是否适合你，也能看出这个团队的技术决策是更自上而下，还是鼓励讨论和迭代。

### 6. 如果后续项目里涉及效果和交付周期冲突，团队通常怎么做取舍？
**适用原因：**

这是一个很成熟的反问。它不是在质疑团队，而是在了解他们面对真实业务压力时的决策风格。也能帮你判断这个团队是不是只追短期 demo，还是会做长期积累。

### 7. 你觉得这个岗位上做得比较好的同学，通常具备哪些特质？
**适用原因：**

这个问题可以让对方主动告诉你“优秀画像”，你也能据此判断自己是否匹配，并顺势补一句自己的优势点。

### 8. 如果后续流程顺利，终面或下一轮你建议我重点准备哪一块？
**适用原因：**

这是非常实用的收尾问题。一方面显得你认真对待这次机会，另一方面也能直接获取下一轮的准备方向。如果对方给出的提示和你目前的准备一致，你也会更有信心。
