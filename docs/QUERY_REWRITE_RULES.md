# Query 改写规则梳理与维护

检索前对用户 query 做改写时，**以梳理好的规则为主、可配置、可维护**；大模型仅作为可选补充（在规则之后做同义扩展），避免「只让大模型理解、黑盒改写」。

---

## 一、规则类别（建议梳理顺序）

| 类别 | 说明 | 配置形式 | 示例 |
|------|------|----------|------|
| **归一化** | 全角→半角、多余空白合并、大小写（英文） | 固定逻辑 + 可选字符映射表 | `　` → 空格，`％` → `%` |
| **纠错 / 错别字** | 常见错别字、业务内常见笔误 | 映射表：错误 → 正确 | `帐号` → `账号`，`登入` → `登录`（若文档用「登录」） |
| **同义词 / 表述统一** | 用户说法与文档用语不一致，统一到文档侧表述 | 映射表：用户词 → 文档词 | `怎么申请` → `如何申请`，`费用` → `收费标准`（按业务定） |
| **业务术语** | 缩写↔全称、俗称↔正式名 | 双向或单向映射 | `API` ↔ `接口`，`FAQ` → `常见问题` |
| **停用词（可选）** | 检索时去掉无信息量词，减少噪音 | 词列表（去停用词后若为空则保留原 query） | `的、了、呢、请问` 等按需 |

规则执行顺序建议：**归一化 → 纠错 → 同义词/术语**；停用词视业务决定是否使用（有时会误伤关键信息）。

---

## 二、配置格式与位置

- **配置文件**：`data/query_rewrite_rules.json`（可复制 `data/query_rewrite_rules.example.json` 为模板）。
- **结构**：同义词/纠错/术语等用「键值对」列出，代码按 key 替换为 value（整词匹配或按分词后替换，见实现）。
- 规则只增不改代码：新增一条就多在配置里加一项，便于评审和回溯。

---

## 三、与 LLM 的分工

- **规则**：负责确定性的归一化、纠错、同义词/术语统一，保证「文档里怎么写、query 就怎么对齐」，可解释、可审计。
- **LLM（可选）**：在规则改写之后，若开启「query 扩展」，再用大模型生成 1～2 个同义问句，与规则改写后的 query 一起送检索；prompt 中可约束「仅做同义改写、不改变专业术语」，必要时在 prompt 里注入业务同义词表，减少乱改。

这样既能把规则梳理清楚、落在配置里，又不排斥在规则之上用大模型做有限扩展。

---

## 四、如何梳理出规则（开发阶段无数据时）

1. **从文档用语反推**：抽一批文档标题/小标题/高频词，列出「官方说法」；用户可能用口语、错别字、同义说法，对这些列映射到官方说法。
2. **从少量典型问题反推**：自己写 20～50 条典型问法，看哪些检索不到或召回差；对这类 query 做一次「若改成 XX 就能命中」的手工改写，把「原词 → 改写词」记下来，沉淀进同义词/纠错表。
3. **迭代**：先上归一化 + 少量纠错/同义词，观察检索与回答效果，再逐步加术语、停用词；有线上数据后再用日志补充规则。

---

## 五、代码接入

- 规则改写实现：`src/kb/query_rewrite.py`（读 `data/query_rewrite_rules.json`，按顺序应用归一化、纠错、同义词）。
- 检索入口：在 `retrieve_with_validation` / `_merge_3_7` 入口对原始 query 先做一次 `rewrite_query_by_rules(query)`，再用改写后的 query 做检索；缓存 key 建议用改写后的 query，避免同一语义多 key。
